# Parallax — input→output flow audit (2026-06-07)

Read-only audit. No code changed. Engine 42/42 pass; built `index.html` is in sync with source.

**Headline:** the engine and the Scenarios lever→override wiring are sound (correct signs, units,
year-base, path-consistency). The damage is on the **INPUT side**: several core engine inputs a real
(non-demo) household literally cannot enter, so every real plan silently runs on defaults.

---

## CRITICAL — engine inputs with NO way to enter them (real plans run wrong silently)

1. **SS claim age — uneditable.** `claimAge` has no input field anywhere; shown as read-only text
   only (`parallax_v2.html:1995-96`). Permanently stuck at 67. The "SS Start Age" lever only sends a
   *delta* off this fixed base. → every household's actual filing age is wrong unless it's 67.
2. **LTC cost — uneditable.** Engine reads `ltc.amount`/`ltc.onsetAge` (`engine.js:761,688`) but no
   input writes them. Set only by the demo seed. → real plans always model **$0 long-term care**.
3. **Housing & debt spending — uneditable.** Engine adds `expenses.housing` + `expenses.debt` to
   spending (`engine.js:764-65`). Expenses page binds only `living` + `healthcare`. → housing is
   invisible/uneditable; `debt` is fully dead (always 0).
4. **Withdrawal strategy — uneditable.** Engine sequences account draws by
   `portfolio.withdrawalStrategy` (`engine.js:832`), no UI surface writes it. Always `taxable-first`.
   Invisible but materially changes tax drag every run.

## DEAD INPUTS — advisor types a value, engine ignores it

5. **`income.workingIncome`** ("Income / yr" in header, `:1964`) — not read by the engine at all
   (confirmed: 0 hits in `engine.js`). Used only for a UI cash-flow display.
6. **`properties[].purchaseYear`** (`:2413`) — not read by the engine (0 hits). Pure dead field.

## ORPHAN ENGINE FIELDS — read by engine, no input populates them (silent defaults)

7. **`savings.split`** — no input; real households' contributions always land 100% Traditional.
8. **`properties[].appreciation` / `.commissionPct`** — no input; sale math always uses 0% / 5%.
9. **`properties[].mortgage.startAge`** — no input; defaults to current age, so an existing mortgage's
   remaining balance/payment is computed wrong.
10. **`income.pension.colaPct`** — no input; every pension modeled flat-nominal (erodes in real terms),
    no way to mark it inflation-linked.

## LATENT BUGS (masked by today's blank/default plan, surface once inputs are filled)

11. **Spending lever scope mismatch.** Lever computes its fraction off `living` only
    (`:1828-31`) but the engine applies that fraction to `living + housing + debt + extras`
    (`engine.js:763-74`). Once housing/debt are non-zero, the realized spend change ≠ the dollar
    figure the slider shows. (Also: a "Lifestyle" slider silently moving housing+debt is questionable
    even if intended.)
12. **Savings lever inert at $0 base.** `savingsBump` is a fraction of base savings; engine does
    `base*(1+bump)` (`engine.js:652`). If base `savings.annual` is 0 (current default), raising the
    slider does nothing. The `/Math.max(1,…)` guard avoids NaN but can't rescue a 0 base.

## INCONSISTENCIES / DRIFT TRAPS

13. **Cash-flow path contradicts ORDERS.md.** The cash-flow drawer renders a Monte-Carlo seeded sim
    `res.sims[cfPathIndex]` (`:3997-99`), but the ORDERS status log (2026-06-06) states it was switched
    to the **deterministic `paths.expected`**. Code now does the opposite (deliberately, per the
    comment) → the doc is stale OR the change was reverted. Reconcile. Consequence: the card's
    "Expected end · today's $" (deterministic, `:3864`) and the cash-flow table's ending balance
    (MC path) **will not match** — confusing in front of a client.
14. **Pension start age: input vs sim disagree.** Income-page `pension.startAge` drives the gutter
    display (`:2225,2606`) but the sim always uses the auto-linked `pensionAge` lever
    (`ov.pensionStartAge = L.pensionAge`, `:1844`). The two views can show different pension timing.
15. **Spouse age stored twice.** `meta.spouseAge` (`engine.js:365`) vs the live
    `household.spouse.currentAge` (what's actually read/edited). `meta.spouseAge` is never synced —
    orphan today, drift trap if anything starts reading it.
16. **Inconsistent age-anchor semantics.** Goals + preset extras shift when you correct the primary's
    current age (`aStart/aEnd` anchoring); `income.other` and `liabilities` rows bind start/end age
    directly and do NOT shift. Same-looking "age 65–75" field behaves differently by row type.
17. **Doc drift:** `CLAUDE.md` says 38 engine tests; actual is **42** (ORDERS.md is correct).

## VERIFIED CORRECT (checked, no bug)

Retirement-age / SS / one-time-event / allocation lever mappings (sign, unit, year-base — no
off-by-one); spend increase→bump / decrease→cut split; per-column goal-amount overrides; path-
consistency (all columns share one seeded bundle); monthly↔annual, pct→fraction, basisPct, taxpct,
realGrowth, taxablePct unit conversions; success ring reads `successRate` directly; "Expected end"
reads `paths.expected`; cash-flow Return$ = `startBalance × realReturnUsed` (no double-count);
Sequencing lines from `runHistoricalPath`, cards from engine row fields.
